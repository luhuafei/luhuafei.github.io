IOS试题

发表于 2017-10-01 | 分类于 试题

从网上看到很多面试试题, 总结一些自己觉的不错的.

为什么说OC是一门动态语言

oc类的类型和数据变量的类型都是在运行时确定的,而不是在编译时确定的,例如多态性: 多态特性,
我们可以使用父类对象来指定子类对象, 并且可以用来调用子类的方法.运行时特性, 我们可以动态
的添加方法或者替换方法.

为什么代理要用weak? 代理的delegate和dataSource有什么区别? block和代理的区别?

代理是用weak修饰的:
   1使用weak是为了避免循坏引用,
   2.当使用weak修饰的属性, 当对象释放的时候, 系统会对属性赋值nil, Object-C有个特性
     就是对nil对象发送消息时, 不会crash.
delgate表示代理, 代理可以让A对象通知B对象, A发生变化, 前提是B遵循了A的代理, 并且实现了A的
方法.
dataSource表示数据源, 如果A对象声明了数据源, 当我们创建A对象的时候,我们就该实现数据源, 来告诉A
他所需要的一些数据.例如:tableview数据源方法, 需要告诉它, 我要实现几组cell, 每组cell返回多少行,
实现的cell什么样子.
delgate和datasource都可以使用require和optional来修饰
代理和block的区别
  相同点:
        1.代理和block大多都可以用来做倒序传值的,
        2.注意避免循坏引用.
        3.调用的时候都要判断是否已经实现.
  不同点:
        1.代理必须先声明方法, block则是保存的一段代码, 其实就是一个函数,
        2.修饰不一样, 代理用weak修饰, block用copy修饰block则是保存的一段代码

为什么IBOutlet修饰的控件使用weak来修饰的?

在xib或者sb中的拓控件时, 其实控件就加载到了父控件的subviews数组里面了, 已经进行了强引用,
即weak, 也不会造成对象的释放.

Runtime

objc在向一个对象发送消息时,发生了什么?

1.根据对象的isa指针找到类对象id, 在查询类对象里面的methodLists方法函数列表, 
 2.如果没有找到, 再沿着superClass寻找父类, 再在父类的methodLists方法列表里查询, 
 3.最终找到SEL, 根据id和 SEL确认IMP(指针函数), 再发送消息.

什么时候会报unrecognized selector错误? ios有哪些机制避免走到这一步?

1.当发送消息的时候, 我们会根据类里面的methodLists列表去查询我们要动用的SEL, 
当查询不到时候, 我们会直沿着父类查询, 

2.当最终查询到的时候我们会报unrecognized selector错误, 当系统查询不到方法的时候,会调用 +  (BOOL)resolveInstanceMethod:(SEL)sel
 动态解释的方法来给我一次机会来添加,调用不到的方法. 

3或者我们可以再次使用- (id)forwardingTargetForSelector(SEL)sSelector                      
 重定向的方法来告诉系统, 该调用什么方法,一来保证 不会崩溃.

能否向编译后得到的类中增加实例变量? 能否向运行时创建的类中添加实例变量? 为什么?

1.不能向编译后得到的类增加实例变量
2.能向运行时创建的类中添加实例变量
解释:
  1.编译后的类已经注册在runtime中, 类结构体中的objc_ivar_list 实例变量的链表和 instance_size实例变量
      的内存大小已经确定, runtime会调用class_setvarlayout 或 class_setWeaklvarLayout来处理strong 和 
      waek引用, 所以不能向存在的类中添加实例变量.
   2,运行时创建的类是可以添加实例变量的, 调用class_addIvar函数, 但是在调用objc_allocateClassPair之后, 
       objc_registerClassPair之前, 原因同上.

runtime如何实现weak变量的自动置nil?

1.runtime对注册的类, 会进行布局, 对于weak对象会放入一个hash表中. 用weak指向的对象内存地址作为 
   key, 当此对象的引用计数为0的时候回dealloc.

 2.假如weak指向的对象内存地址是A, 那么就会以A为键, 在这个weak表中搜索, 找到所有以A为键的weak
    对象, 从而置为nil.

给类添加一个属性后, 在类结构体里那些元素会发生变化

1.instance_size: 实力内存的大小
2.objc_ivar_list *ivars : 属性列表

首先Runtime 系统会把方法调用转化为消息发送, 即objc_msgSend, 并且把方法的调用者, 和方法选择器, 当做参数传递过去,此时, 方法的调用者会通过isa指针来找到其所属的类, 然后cache 或者 methodLists中查找该方法, 找到就跳到对应的方法去执行.
如果在类中没有找到该方法, 则通过super_class往上一级超类查找(如果一直找到NSObject都没有找到该方法, 这种情况, 我)

混淆点:

 @implementation Son : Father
- (id)init
{
    self = [super init];
    if (self)
    {
        NSLog(@"%@", NSStringFromClass([self class]));
        NSLog(@"%@", NSStringFromClass([super class]));
    }
    return self;
}
@end

常见的例子, 答案输出的全部是Son
NSStringFromClass([self class])这个为什么是Son, 不做分析啦都能理解, 下面细分析一下 NSStringFromClass([super class])这个的内部的实现,
1.第一步先构造objc_super结构体, 结构体的第一个成员就是self, 第二个成员是
(id)class_getSuperclass(objc_getClass(“Son”)).
2.第二步是去father这个类里去找-(Class)class, 没有, 然后去NSObject类去找, 找到了, 最后内部是使用
objc_msgSend((objc_super->receiver) ,@selector(class))去掉用, 此时已经和[self class]调用的相同了, 所以两个
输出的结果都是Son.

更多相关 runtime

runLoop

runloop是用来做什么的? runloop和线程有什么关系? 主线程默认开启了runloop吗? 子线程呢?

runloop:循坏跑圈,用来处理线程里面的事件和消息.
runloop和线程的关系:每个线程如果想继续运行, 不被释放, 就必须有一个runloop来不停地跑圈, 处理线
                                  程里面的各个事件和消息
主线程默认开启一个runloop, 也就是这个runloop才能保证我们程序正常的运行, 子线程是默认没有开始
runloop

runloop的model是用来做什么的? 有几种mode

系统默认注册了5个model:
      1.KCFRunLoopDefaultMode:App的默认Mode, 通常主线程是在这个Mode下运行的.
      2.UITrackingRunLoopMode: 界面跟踪Mode, 用于ScrollView追踪触摸、混动, 保证界面滑动时不受
          model影响
      3.UIInitializationLoopMode:在刚启动App时,进入的第一个mode, 启动完成后就不再使用啦
      4,GSEvenReceiveRunLoopMode:接受系统事件的内部model, 通常用不到
      5,KCFRunLoopCommonModes:这是一个展位的mode, 没有实际作用.
      iOS对以上的mode进行了封装: NSDefaultRunLoopMode  和   NSRunLoopCommonModes

为什么吧NStimer对象以NSDefaultRunLoopMode添加到主运行循坏以后, 滑动scrollView的时候NSTimer却不动了

NStimer对象是在NSDefaultRunLoopMode下面调用消息的, 但是当我们滑动scrollview的时候,
NSDefaultRunLoopMode      
模式就自动切换到UITrackingRunLoopMode模式下面, 却不可以继续响应NSTimer发送消息,所以如果想在滑动
scrollView的情况下面还调用NSTimer的消息, 我们可以把NSRunloop的模式更改为NSRUnLoopCommonModes

什么是arc?(arc解决什么问题?)

首先解释ARC:automatic reference counting 自动引用计数, ARC几个要点: 在对象被创建时retain count +1, 在对象  
被release时, retain count -1, 当 retain count 为0时, 销毁对象,
程序员中加入autoreleasepool的对象会由系统自动加上autorelease方法, 如果该对象引用计数为0, 则销毁.

解决问题; MRC下内存管理的缺点,
  1.当我们要释放一个堆内存时, 首先要确定只想这个堆空间的指针都没release了(避免提前释放).
  2.释放指针指向的堆空间, 首先确定哪些指针指向同一个堆, 这些指针只能释放一次.(MRC下即谁创建水释放, 避
     免重复释放)       
  3.模块化操作时, 对象可能被多个模块创建和使用, 不能确定最后由谁去释放
  4.多线程操作时, 不确定那个线程最后使用完毕.

请解释以下keywords 的区别: assign VS weak, block VS weak

assin使用于基本数据类型, weak是使用于NSobject对象, 并且是一个弱引用.
assign其实也可以用来修饰对象, 那么我们为什么不用它呢?  因为被assign修饰的对象在释放之后, 指针的地址
还是存在的, 也就是说指针并没有被置nil, 如果后续的内存分配中, 刚好分到了这块地址, 程序就会崩掉. 而weak
修饰的对象在释放后, 指针地址会被置nil, 所以现在一般弱引用就是用weak.

首先__block是用来修饰一个变量, 这个变量就可以在block中被修改
   __block;使用__block修饰的变量在block代码块中会被retain(ARC下, MRC下不会retain)
   __weak:使用__weak修饰的变量不会在block代码块中被retain.
  同时, 在ARC下, 要避免block出现循坏引用__weak rypedof(self)weakself = self;

+(void)load; +(void)initialize; 有什么用处?

在Objcetive-C中, runtime会自动调用每个类的两个方法. +(void)load会在类初始加载时调用, +(void)initialize会在
第一次调用类的类方法或实例方法之前被调用, 这两个方法是可选的, 且只有在实现了他们时才会被调用
共同点:两个方法都只会被调用一次.
